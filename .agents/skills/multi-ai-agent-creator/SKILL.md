# Multi-AI Agent Creator

## Overview

Create one concise source of truth for AI coding-agent behavior, then adapt it into the files each target tool is likely to consume. Prefer minimal, repo-specific constraints over generic model coaching.

## Workflow

1. Inspect the repository before writing instructions.
   - Find existing `AGENTS.md`, `.cursor/rules/*.mdc`, `.cursorrules`, `CLAUDE.md`, `CONTRIBUTING.md`, package manifests, test config, CI workflows, and docs.
   - Preserve user-authored rules unless they conflict with the requested update.
   - Treat commands in package manifests and CI as stronger evidence than README prose.

2. Decide the targets.
   - Default to `codex,opencode,cursor,devin,antigravity` when the user says "all my AIs" or names the tools from the initial request.
   - Read `references/target-mapping.md` when choosing file paths or when a target-specific convention matters.
   - For any target whose official file convention is unknown or unstable, create a clearly labeled manual-import file under `.ai/` instead of inventing an auto-loaded path.

3. Generate the pack.
   - Use `scripts/create_multi_ai_pack.py` when a repo-level instruction pack is needed.
   - Run from the target repository:

```bash
python path/to/multi-ai-agent-creator/scripts/create_multi_ai_pack.py --repo . --out . --targets codex,opencode,cursor,devin,antigravity
```

4. Edit the generated files by hand when project nuance matters.
   - Add only instructions that future agent sessions cannot cheaply infer from the repo.
   - Keep commands exact and include the working directory when it is not the repo root.
   - Prefer constraints, safety rails, verification expectations, and known gotchas over broad style advice.
   - Do not include secrets, private credentials, local-only paths, or hidden chain-of-thought requirements.

5. Validate.
   - Check that generated files are concise and non-contradictory.
   - Confirm referenced commands exist in manifests, CI, or project docs.
   - If files were generated by script, run with `--check` after edits to confirm expected target files exist.

## Output Conventions

Use these defaults unless the user asks for a different layout:

- Shared agent profile: `.ai/agent-profile.md`
- Codex and opencode project rules: `AGENTS.md`
- Cursor project rule: `.cursor/rules/project.mdc`
- Devin handoff note: `.ai/devin-instructions.md`
- Antigravity handoff note: `.ai/antigravity-instructions.md`

When a repository already has a stronger convention, adapt to it. For example, update an existing `.cursor/rules/*.mdc` file instead of creating a duplicate rule file.

## Instruction Quality Bar

Good multi-agent instructions are:

- Specific to this repo.
- Short enough to be read on every session.
- Written as operational guidance: commands, boundaries, review expectations, risk areas.
- Split by scope only when there are genuine subprojects, packages, or generated/vendor directories.
- Honest about uncertainty: use "verify with docs" or "manual import" where needed.

Avoid:

- Persona theater that does not change behavior.
- Huge pasted docs that should stay linked or lazily loaded.
- Duplicate rules across files that can drift.
- Commands copied from memory when the repo contains the real command.
- Tool-specific claims that have not been verified against current docs.

## Updating Existing Packs

When a pack already exists, update the source-of-truth content first, then regenerate or mirror target-specific files. If the repo has separate instructions per package, preserve the hierarchy and add only the smallest root-level note needed to route agents to the right file.
